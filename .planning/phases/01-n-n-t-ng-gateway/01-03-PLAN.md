<objective>
Scaffold the four new stub services (`connection`, `feed`, `search`, `notification`) by cloning the canonical `services/profile-service` skeleton (created by Plan 02) and trimming each to a single `/health` route backed by a real DB ping (D-03, D-08). Each stub `/health` also ECHOES the inbound `X-Request-Id` header back as a `rid` field in its JSON body — this is the concrete downstream-receipt proof for D-12 (ISSUE-5): the smoke test forces the gateway to fan out to each stub, then asserts the gateway-forwarded request id actually arrived at a downstream service. Create the four matching gateway typed clients (D-11). This plan owns ONLY the four new service directories and the four new client files — it does NOT touch docker-compose.yml, routes.php, public/index.php, or db/* (Plan 04 wires compose + DI; Plan 05 provisions DBs).

This plan runs in Wave 2 and `depends_on: [02]` because it clones from `services/profile-service`, which only exists after Plan 02's `git mv` completes. This avoids the TOCTOU race that would occur if Plan 03 ran in parallel with Plan 02's rename.

Purpose: PLAT-01 (new microservices, each with own DB) + PLAT-05 (clean boot, ≤9 containers) + D-12 downstream X-Request-Id receipt proof. D-08 = minimal Slim app, /health with Db::ping(); D-09 = empty schema (provisioned in Plan 05, not here); D-11 = typed client per service.
Output: 4 stub service trees + 4 gateway clients.
</objective>

public function health(Request $req, Response $res): Response {
    $dbOk = \App\Db::ping();
    return \App\Json::raw($res, [

        'db'     => $dbOk ? 'ok' : 'down',

        'ts'     => gmdate('c'),
