<objective>
Rewrite DB provisioning for ProConnect (D-01, D-06, D-09, D-13): five `proconnect_*` schemas, five scoped `*_svc` users, the profile users table + 5 reseeded demo accounts, an idempotent live-volume cutover migration template, AND the removal of the retired blog post/comment schema files so a fresh init dir cannot recreate them. This plan owns ONLY the `db/` files — it does NOT touch docker-compose.yml (the 5 new `*_SVC_DB_PASS` env vars in the mariadb block are added by Plan 04) or any service code.

CRITICAL (RESEARCH Pitfall 1): `db/00-init.sh` runs ONLY on a fresh volume. The live VPS volume already exists, so the init script alone will silently no-op there. Therefore we ship BOTH: (1) updated init scripts for fresh environments, and (2) `db/migrate-phase1.sql.tmpl` for the existing live volume, applied to the running container via envsubst.

CRITICAL (ISSUE-1 — fresh-volume init runs ALL db/*.sql alphabetically): MariaDB's `/docker-entrypoint-initdb.d` mount is `./db` (see docker-compose.yml line 18). On a fresh volume it executes EVERY `db/*.sql` (and `*.sql.gz`/`*.sh`) in alphabetical order. The current repo still contains `db/02-schema-posts.sql` (USE blog_posts; CREATE TABLE posts) and `db/03-schema-comments.sql` (USE blog_comments; CREATE TABLE comments). If left in place, a fresh `docker compose up` would recreate blog_posts/blog_comments — directly contradicting D-01 (retire post/comment) and the clean 8-container model. Therefore this plan MUST `git rm` those two files. (Their service-folder code is removed by Plan 04; the schema files are removed here because they live in the init mount.)

CRITICAL (ISSUE-7 — live cutover script MUST NOT be in the initdb glob): the live-volume cutover is an envsubst TEMPLATE containing `${*_SVC_DB_PASS}` placeholders and `ALTER USER ... IDENTIFIED BY '${...}'`. If it were named `*.sql`, MariaDB's initdb glob would auto-run it on a FRESH volume WITHOUT envsubst, setting service DB-user passwords to the LITERAL string `${PROFILE_SVC_DB_PASS}` etc. — breaking auth on every fresh environment. Idempotency does NOT save it (the placeholders are never substituted during initdb). Therefore the cutover file is named `db/migrate-phase1.sql.tmpl`: the `.tmpl` extension is semantically correct (it IS an envsubst template) AND the MariaDB initdb glob matches `*.sql`/`*.sql.gz`/`*.sh` but NOT `*.sql.tmpl`, so it will never auto-run on fresh volumes. It is applied ONLY by Plan 06's deploy step via envsubst against the running container.

Purpose: PLAT-01 (database-per-service) + PROF-01 (demo accounts survive so login works). D-01 retire post/comment schema; D-06 wipe & reseed; D-09 empty stub schemas (no business tables); D-13 naming.
Output: `db/00-init.sh` (rewritten), `db/01-schema-profile.sql` (renamed from 01-schema-users.sql), `db/02-schema-posts.sql` (REMOVED), `db/03-schema-comments.sql` (REMOVED), `db/99-seed.sql` (rewritten to proconnect_profile only), `db/migrate-phase1.sql.tmpl` (new — NOT in initdb glob).
</objective>
