with typed feed-service calls, add FeedController whose `feed()` composes THREE services into one
newest-first timeline (connection → feed → profile): it resolves repost originals, then enriches

asks connection (who are my friends) + feed (posts + counts) + profile (authors) → merges 1
timeline → degrades if a part fails." It reuses the verified AggregateController settle/degrade +
ConnectionsController enrich/allowlist patterns; counts come precomputed from feed-service (no
gateway N+1); repost originals are batch-resolved (no N+1) and their authors are folded into the
SAME single profile batch as the post authors.

Output: FeedClient (extended), FeedController, gateway routes + DI.

THE COMPOSITION (04-RESEARCH Pattern 1, lines 144-223) — FeedController::feed.
DESIGN (ONE profile batch — the union-batch shape; originals resolve BEFORE the author batch):
  STEP 1 author universe: `$authorIds=[$me]`; try ConnectionClient::listAccepted($me); 200 → append each
    `user_id`; non-200 or GuzzleException → `$degraded[]='connections'` (own-posts-only). unique.
  STEP 2 posts+counts (HARD dep / spine): FeedClient::timeline($authorIds,$me,$limit); non-200 →
    `Json::raw($res, decode, status)` (fail the feed — feed-service is the only hard dependency).
  STEP 3 repost originals (NO N+1): collect `repost_of` ids (int>0, unique) from STEP 2 rows; if any →
    `Utils::settle([ $this->feed->getPostsAsync($originalIds) ])->wait()` to resolve originals; 200 → map
    originals by id; non-200/GuzzleException → `$degraded[]='reposts'` (originals map empty).
  STEP 4 author enrichment (ONE batch over the UNION): collect `$authorUnion = array_values(array_unique(
    [...post author_ids, ...resolved-original author_ids]))` (int>0); ONE `ProfileClient::batch($authorUnion)`;
    on 200 → build `id→allowlisted card` map (covers BOTH post authors and original authors); on
    non-200/GuzzleException → `$degraded[]='profiles'` (cards null). NOTE the ordering dependency: STEP 4
    runs AFTER STEP 3 resolves, because the union needs the original authors' ids. ≤2 upstream round trips
    after the spine (originals, then the union author batch).
  STEP 5 assemble (preserve feed-service newest-first order — map in place, NEVER re-sort by a map, Pitfall 6):
    each post gets `author` (allowlisted card|null from the union map); if repost_of>0 and original present,
    attach `original` (the original + its allowlisted `author`, also from the union map); else `original:null`
    (deleted original — Pitfall 5).
  Emit `Json::list($res,$out, $degraded? ['degraded'=>true,'parts'=>array_values(array_unique($degraded))] : [])`.
  PARALLELISM (A7, honest): STEP 1 must precede STEP 2 (need author ids); STEP 3 (originals) must precede
    STEP 4 (union author batch) because the union includes the originals' authors. So the fan-out is
    sequential — originals → union author batch — NOT a parallel settle of profile+originals. This is the
    honest shape: ONE profile batch, ≤2 round trips. The `Utils::settle([...])` wrapper around STEP 3 keeps
    the async-client idiom (and trivially extends if more parallel-independent fetches are added), but do
    NOT claim the profile batch runs in parallel with the originals fetch — it cannot, by construction.
