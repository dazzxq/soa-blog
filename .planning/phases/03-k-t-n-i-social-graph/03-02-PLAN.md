`use App\Db; use App\DomainError; use App\Json; use PDOException;` and the PSR-7 aliases.

(1) Private `statusBetween(int $viewer, int $target): string` — the ONE status computation,
reused by status(), and (in Plan 03) by the gateway invariant. Exact logic:
```php
if ($viewer === $target) return 'self';
$stmt = Db::pdo()->prepare(
  'SELECT requester_id, addressee_id, status FROM connections
    WHERE (requester_id = :v1 AND addressee_id = :t1)
       OR (requester_id = :t2 AND addressee_id = :v2) LIMIT 1');
$stmt->execute([':v1'=>$viewer, ':t1'=>$target, ':t2'=>$target, ':v2'=>$viewer]);
$row = $stmt->fetch();
if ($row === false) return 'none';
if ($row['status'] === 'accepted') return 'connected';
return ((int)$row['requester_id'] === $viewer) ? 'pending_outgoing' : 'pending_incoming';
```
The `LIMIT 1` is now provably deterministic: Plan 01's `uq_pair` unique key guarantees at
most one row per unordered pair, so the bidirectional OR can never match two rows.

(2) `status(Request $req, Response $res): Response` — reads `viewer`/`target` from
`$req->getQueryParams()` (ints), returns `Json::ok($res, ['status' => $this->statusBetween($viewer, $target)])`.
This is the D-05 payoff route — the `{"data":{"status":...}}` shape is mandatory.

(3) `create(Request $req, Response $res): Response` (CONN-01) — requester from
`X-User-Id` header (int), addressee from body `addressee_id` (int). Validate:
requester>0, addressee>0, requester !== addressee else DomainError(400,'VALIDATION_FAILED',
'Yêu cầu kết nối không hợp lệ.'). INSERT `(requester_id, addressee_id, status) VALUES (:r,:a,'pending')`;
catch PDOException '23000' → DomainError(409,'REQUEST_EXISTS','Đã tồn tại lời mời giữa hai người.').
NOTE: the '23000' catch now backstops BOTH a same-direction duplicate (uq_conn_pair) AND an
opposite-direction concurrent invite (uq_pair) — one catch covers both races.
Return `Json::ok($res, ['id'=>(int)Db::pdo()->lastInsertId(),'status'=>'pending'], 201)`.

(4) `accept(Request $req, Response $res, array $args): Response` (CONN-02) — caller from
X-User-Id, id from `$args['id']`. Scoped UPDATE
`SET status='accepted' WHERE id=:id AND addressee_id=:caller AND status='pending'`.
Prove via scoped SELECT `WHERE id=:id AND addressee_id=:caller AND status='accepted' LIMIT 1`;
if fetch()===false → DomainError(404,'REQUEST_NOT_FOUND','Không tìm thấy lời mời để chấp nhận.')
(uniform 404 — a non-addressee caller hits this same 404, no oracle). Return Json::ok 200 {id,status:'accepted'}.

(5) `deleteRequest(Request $req, Response $res, array $args): Response` (reject + cancel,
both DELETE pending per D-03). caller from X-User-Id, id from args. DELETE scoped so the
caller may be EITHER party of a pending row:
`DELETE FROM connections WHERE id=:id AND status='pending' AND (requester_id=:c1 OR addressee_id=:c2)`
(bind caller twice). `rowCount()===0` → DomainError(404,'REQUEST_NOT_FOUND','Không tìm thấy lời mời.').
Return Json::ok 200 {deleted:true}. (The gateway in Plan 03 routes /reject and DELETE /requests/{id}
both here; the scope guarantees only requester-or-addressee can delete a pending edge. This
"either pending party deletes" design is INTENTIONAL and reviewer-approved.)

(6) `removeEdge(Request $req, Response $res, array $args): Response` (remove accepted, D-03,
either side). caller from X-User-Id, otherUserId from `$args['userId']`. DELETE
`WHERE status='accepted' AND ((requester_id=:c1 AND addressee_id=:o1) OR (requester_id=:o2 AND addressee_id=:c2))`.
rowCount()===0 → DomainError(404,'CONNECTION_NOT_FOUND','Không tìm thấy kết nối.'). Return 200 {deleted:true}.

NEVER read requester_id/user_id from the body (IDOR). All scoping via X-User-Id only.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | Elevation/Tampering (IDOR) | accept/reject/cancel/remove | mitigate | Every mutation scoped by X-User-Id in the WHERE (addressee for accept; requester-or-addressee for delete-pending; either side for remove). | T-03-09 | Information disclosure | list/suggestion responses | mitigate | connection-service returns only {user_id, status, request_id, direction} — never profile/email fields; enrichment + allowlist happen at the gateway (Plan 03). |
